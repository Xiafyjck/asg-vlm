# cz 运行环境说明

以下是 cz 运行环境说明,内容包括机器与网络拓扑、离线资源策略。可以为每一个需要部署在 cz 环境的项目中加入该文档，增强 agent 的上下文。

## 机器与网络拓扑

理想情况下,mac 应当直接 ssh 到训练服务器,在训练机上原地开发和调试。但训练服务器受管控:它没有外网出口,也不向任何机器暴露 ssh 入口,mac 连不上它,集群内的其他服务器同样连不进去。幸好训练服务器挂载着一个共享卷,集群内一台能上网的普通服务器也挂载着同一个卷。于是实际拓扑绕道而行:mac ssh 到这台普通服务器,训练服务器与外界的全部交互都经共享卷完成——普通服务器把资源写进共享卷供训练服务器读,训练服务器把训练产物写回共享卷供普通服务器取走、上传。

本环境由三台机器组成:

- **mac**:客户端。写代码、发起 ssh、运行 Claude Code。
- **4090**:带一张 RTX 4090 的服务器,运行 ubuntu 22.04 docker 容器,CUDA 12.8。可以访问公网,但没有静态 IP,mac 通过 wireproxy 建立的隧道 ssh 进来。它是整套环境唯一的对外入口,也是 h200 与外界之间唯一的中转,承担全部需要网络的工作:安装 python 环境、下载模型权重等资源、上传训练产物、补传实验日志;也可以用单卡跑小规模调试。
- **h200**:带八张 H200 的服务器,运行与 4090 相同的 ubuntu 22.04 镜像,CUDA 12.8,与 4090 处于同一个环境。它受管控,不接受任何 ssh 连入,与外界交换数据只靠共享卷。它只做训练。

连接关系:

```
mac ──wireproxy + ssh──> 4090 <──共享卷──> 8×H200
```

这套拓扑要正常运转,依赖以下三条约定:

- 4090 与 h200 挂载同一个共享卷,且两边挂载路径必须完全一致:虚拟环境和脚本里记录的都是绝对路径,路径不一致则全部失效。
- h200 需要的一切资源(python 环境、模型权重等)都由 4090 预先放进共享卷。h200 从不联网,任何联网尝试都应配置成显式报错,而不是卡在超时。
- Claude Code 在 mac 上通过 ssh 连到 4090 干活:访问 AI 服务的流量发生在 mac 一侧,服务器本身不需要梯子。

两台服务器的 GPU 架构不同,会影响带编译产物的依赖:4090 是 Ada(sm_89),H200 是 Hopper(sm_90)。同一个虚拟环境两机共用没有问题,但 GPU 架构相关的编译产物(典型:flash-attn 的内核)必须同时覆盖两种架构——官方预编译轮子内含 sm80 与 sm90 内核,两机都能跑(sm_89 可执行 sm80 内核);若源码自编,先设 `TORCH_CUDA_ARCH_LIST="8.9;9.0"`。deepspeed 这类运行时 JIT 编译的包在各自机器首跑时本地编译,不需要网络,但镜像里要有编译工具链。

## 离线资源策略

linux 有一套 XDG Base Directory 规范,约定了缓存、配置、数据各自的标准存放位置,并规定这些位置可以用环境变量重定向。大部分软件遵守这套规范:默认把缓存写在用户主目录下的标准位置,也就能被环境变量改到任意别处。把这些位置指到共享卷,就实现"4090 下载,h200 直读"。目前确认支持缓存重定向的工具有 **uv、huggingface、modelscope**。
